# Contributing to SecureTrack

First off, thank you for considering contributing to SecureTrack! It's people like you that make this tool such a powerful defense system.

## Code of Conduct

By participating in this project, you are expected to uphold our Code of Conduct:
-   **Be Respectful**: Treat everyone with respect and dignity.
-   **Be Constructive**: Critique ideas, not people.
-   **No Harassment**: Harassment of any kind will not be tolerated.

## How Can I Contribute?

### Reporting Bugs
This section guides you through submitting a bug report for SecureTrack. Following these guidelines helps maintainers and the community understand your report, reproduce the behavior, and find related reports.

-   **Use the Template**: When opening a new issue, please use the provided Bug Report template.
-   **Be Specific**: Include version info, device model, and Android version.

### Suggesting Enhancements
This section guides you through submitting an enhancement suggestion for SecureTrack, including completely new features and minor improvements to existing functionality.

-   **Use the Template**: When opening a new issue, please use the provided Feature Request template.

## Pull Requests

1.  **Fork the Repo**: Create your own fork of the code.
2.  **Create a Branch**: `git checkout -b feature/AmazingFeature`
3.  **Commit Your Changes**: `git commit -m 'feat: Add some AmazingFeature'`
    -   *Please use Conventional Commits (feat, fix, docs, style, refactor).*
4.  **Push to the Branch**: `git push origin feature/AmazingFeature`
5.  **Open a Pull Request**: Submit your PR and describe the changes detailedly.

## Styleguides

### Git Commit Messages
-   Use the present tense ("Add feature" not "Added feature").
-   Use the imperative mood ("Move cursor to..." not "Moves cursor to...").
-   Limit the first line to 72 characters or less.
-   Reference issues and pull requests liberally after the first line.

### Kotlin Style
-   Follow the [Official Kotlin Coding Conventions](https://kotlinlang.org/docs/coding-conventions.html).
-   Keep UI logic in Activities/Fragments and business logic in ViewModels/Services.
-   Use descriptive variable names.

Thank you for your contributions!
